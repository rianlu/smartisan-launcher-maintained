# 旧桌面图标来源统一设计

## 背景

当前旧桌面已经具备以下基础能力：

- 通过 `redirect_icons` 为单个应用存储替换后的图标字节
- 通过现有刷新链路同步桌面与搜索结果
- 通过旧设置页在“默认图标 / 改进版图标”之间做切换

但在接入第三方图标包后，旧模型已经不够表达真实状态。当前问题主要有三类：

- 页面仍按二选一 UI 展示，无法正确承载“改进版 + 图标包 + 默认图标”
- `useImprovedAppIcon` 仍被当作最终状态使用，导致“当前图标包未适配却仍显示勾选”
- 顶部和列表文案混用了“状态”和“动作”，用户难以理解

本设计目标不是继续在旧布尔模型上打补丁，而是把旧桌面的图标能力整理为统一的来源模型，并在此基础上支持第三方图标包与单应用覆盖。

## 目标

- 支持独立选择“当前图标包”
- 支持全局规则：`改进版图标 > 当前图标包 > 默认图标`
- 支持单应用永久覆盖全局规则，直到用户主动重置
- 支持列表页展示每个应用的当前生效来源
- 支持批量重算并落库，不侵入桌面渲染主链路
- 支持旧数据兼容，不要求一次性清库

## 非目标

- 不支持本地图片自定义图标
- 不支持动态图标、动态时钟、动态日历
- 不支持 `iconback` / `iconmask` / `iconupon` 合成
- 不支持在 V1 中引入新的系统级图标服务
- 不追求兼容所有非标准 icon pack 格式

## 核心结论

### 1. 图标能力分为两层

- 全局规则
- 单应用覆盖

这两层必须明确分开，否则页面和数据都会混乱。

### 2. 最终优先级

最终渲染优先级统一为：

`单应用覆盖 > 改进版图标 > 当前图标包 > 默认图标`

解释如下：

- 单应用覆盖用于表达“用户明确对某个 app 做了单独设置”
- 改进版图标仍然是锤子内置风格，作为全局第一优先级保留
- 当前图标包作为全局第二优先级，为改进版未命中的应用补位
- 默认图标是最终兜底

### 3. 单应用设置采用主流第三方桌面心智

单应用设置默认是 `跟随全局`。

当用户手动修改单个应用的图标后，该应用进入 `单独设置` 状态，并持续覆盖全局规则，直到用户手动选择 `跟随全局`。

这与 Smart Launcher、Niagara 等主流桌面的交互心智一致。

## 产品模型

## 全局设置

设置页首页应包含三个清晰入口：

- `优先使用改进版图标`：全局开关
- `设置图标包`：进入图标包选择页
- `应用图标`：进入全量应用列表页

说明：

- `优先使用改进版图标` 只控制全局链路中的第一优先级是否启用
- `设置图标包` 只负责选择当前图标包，不直接承载批量应用或来源说明
- 原 `改进版图标` 页面应改名为 `应用图标` 或 `图标管理`，避免继续沿用过时语义

### 图标包选择页

图标包选择页只负责一件事：设置当前图标包。

候选项：

- `不使用图标包`
- 已安装并成功解析的第三方图标包列表

命名要求：

- 不使用 `默认图标包`
- 统一使用 `不使用图标包`

原因：

- “默认图标包”容易与“默认图标”混淆
- 这里表达的是“当前没有启用任何第三方图标包”

## 全局生效规则

### 当“优先使用改进版图标”开启时

对每个未被单独设置的应用按以下顺序求值：

1. 若存在改进版图标，使用改进版图标
2. 否则若当前图标包命中，使用图标包图标
3. 否则使用默认图标

### 当“优先使用改进版图标”关闭时

对每个未被单独设置的应用按以下顺序求值：

1. 若当前图标包命中，使用图标包图标
2. 否则使用默认图标

## 单应用覆盖规则

每个应用的最终模式分为两大类：

- `跟随全局`
- `单独设置`

### 单独设置支持的动作

V1 建议支持：

- `跟随全局`
- `使用默认图标`
- `使用改进版图标`
- `使用当前图标包图标`
- `从图标包中另选图标`

其中：

- `跟随全局` 表示清除单应用覆盖
- `使用默认图标` 表示强制始终使用应用原始图标
- `使用改进版图标` 表示强制使用锤子改进版图标
- `使用当前图标包图标` 表示强制使用当前图标包为该应用提供的默认匹配图标
- `从图标包中另选图标` 表示从当前图标包资源中选择一个非默认匹配图标，后续版本可做

### 单应用优先级

一旦某个应用存在单独设置，则该应用不再受以下动作影响：

- 开关“优先使用改进版图标”
- 切换当前图标包
- 批量应用全局规则

只有用户手动选择 `跟随全局`，该应用才重新回到全局优先级链路中。

## 页面设计

## 设置首页

首页推荐结构：

1. `优先使用改进版图标` 开关
2. `设置图标包`
   - 副标题显示：当前图标包名称 / 不使用图标包
3. `应用图标`
   - 进入全量应用列表

不建议继续在首页放“应用当前图标包 / 恢复默认图标”这类混合按钮。

## 应用列表页

列表页应显示所有 launcher apps，而不是仅显示 `redirect_icons` 中已有记录。

每个应用建议展示：

- 应用名
- 当前生效图标预览
- 当前来源说明
- 是否为单独设置

### 状态文案建议

对用户展示的文案应统一表达“当前生效来源”，示例：

- `当前使用：改进版图标`
- `当前使用：图标包`
- `当前使用：默认图标`
- `单独设置：默认图标`
- `单独设置：改进版图标`
- `单独设置：图标包`

如果当前图标包未命中，但该应用仍在跟随全局，则应展示：

- `当前使用：默认图标`
或
- `当前使用：改进版图标`

而不是继续显示“已勾选图标包但当前未适配”。

### 列表交互

点击应用后弹出操作菜单，菜单项顺序建议如下：

1. `跟随全局`
2. `使用默认图标`
3. `使用改进版图标`
4. `使用当前图标包图标`
5. `从图标包中另选图标`

V1 如需控制复杂度，可先实现前四项，把第五项挂为后续增强。

## 文案规范

以下文案需要明确区分：

- `跟随全局`
  表示清除单应用覆盖
- `使用默认图标`
  表示强制原始图标
- `恢复全部默认`
  只用于全局重置场景

禁止继续把 `恢复默认图标` 同时用于“单应用强制默认”和“清除单应用覆盖”。

## 数据模型

## 总体原则

继续复用 `redirect_icons`，不新建数据表。

但 `useImprovedAppIcon` 不能再被当作最终来源判断，而只能作为兼容字段或物化状态的一部分。

真正的来源判断应迁移到来源类型字段。

## `RedirectIconInfo` 的来源类型

建议将现有 `iconSourceType` 明确收口为以下值：

- `global_improved`
- `global_pack`
- `force_default`
- `force_improved`
- `force_pack`
- `force_pack_custom`

说明：

- `global_improved`
  表示该应用当前图标由全局规则计算后落为“改进版图标”
- `global_pack`
  表示该应用当前图标由全局规则计算后落为“图标包图标”
- `force_default`
  表示该应用被单独设置为默认图标
- `force_improved`
  表示该应用被单独设置为改进版图标
- `force_pack`
  表示该应用被单独设置为当前图标包默认匹配图标
- `force_pack_custom`
  表示该应用被单独设置为图标包中的自选图标

### 相关字段映射

- `iconSourceType`
  存来源类型
- `iconSourcePackage`
  存图标包包名，仅 `global_pack` / `force_pack` / `force_pack_custom` 使用
- `iconSourceDrawable`
  存命中的 drawableName，仅 pack 类来源使用
- `useImprovedAppIcon`
  不再直接表达“是否使用改进版”
  新语义调整为“当前是否使用非默认图标的物化结果”

### 物化策略

旧桌面渲染链路已经依赖 `iconData`，因此 V1 仍采用“先计算最终结果，再落库”的物化方案。

也就是说：

- 跟随全局的应用，不在渲染时动态判断来源
- 当全局规则变化时，统一重算所有跟随全局的应用，并将结果批量写回 `redirect_icons`
- 单独设置的应用不参与这次重算

这样可以继续复用现有桌面刷新逻辑，不侵入显示主链路。

## 全局设置持久化

建议保留并正式使用以下全局配置：

- `current_icon_pack_package`
- `improved_icons_enabled`

其中：

- `current_icon_pack_package`
  记录当前选中的图标包包名；为空表示不使用图标包
- `improved_icons_enabled`
  记录“优先使用改进版图标”开关状态

现有 `prefs_key_current_icon_pack` 可继续沿用。

## 关键数据流

## 进入图标包选择页

1. 扫描已安装应用
2. 识别并解析可支持的 icon pack
3. 读取 `current_icon_pack_package`
4. 高亮当前选择
5. 用户选择后保存当前图标包

仅保存选择，不在该页面直接批量应用。

## 全局规则变化

触发条件包括：

- 切换“优先使用改进版图标”
- 更换当前图标包
- 当前图标包卸载

处理流程：

1. 获取所有 launcher apps
2. 过滤掉存在单应用覆盖的应用
3. 对剩余应用按全局规则重新求值
4. 对求值结果为改进版或图标包的应用写入物化记录
5. 对求值结果为默认图标的应用清理物化记录
6. 批量通知 launcher 与搜索刷新

## 单应用设置变化

### 选择“跟随全局”

1. 删除或清空该应用的单应用覆盖记录
2. 按当前全局规则立即重新计算该应用
3. 若结果为改进版或图标包，则写入对应 `global_*` 记录
4. 若结果为默认图标，则移除该应用重定向记录
5. 通知桌面刷新该应用

### 选择“使用默认图标”

1. 写入或更新 `force_default`
2. 清空 `iconData`
3. 清空 pack 来源字段
4. 标记该应用不再跟随全局
5. 通知桌面刷新该应用

### 选择“使用改进版图标”

1. 获取该应用改进版图标
2. 生成 `iconData`
3. 写入 `force_improved`
4. 通知桌面刷新该应用

### 选择“使用当前图标包图标”

1. 基于当前图标包查该应用默认匹配图标
2. 若未命中，阻止操作并给出“当前图标包未适配”
3. 若命中，生成 `iconData`
4. 写入 `force_pack`
5. 记录 `iconSourcePackage` 和 `iconSourceDrawable`
6. 通知桌面刷新该应用

## 旧数据兼容

## 问题

当前库里已经存在以下旧状态：

- 仅有 `useImprovedAppIcon`
- `iconSourceType = pack`
- 部分记录由全局批量应用产生，部分记录由单应用操作产生

旧模型无法可靠区分“这是全局物化结果”还是“这是单应用永久覆盖”。

## 兼容策略

V1 采用保守兼容方案：

1. 已有 `pack` 记录统一视为旧版全局图标包物化结果，迁移到 `global_pack`
2. 已有非 `pack` 且 `useImprovedAppIcon = true` 的记录，统一视为旧版改进版物化结果，迁移到 `global_improved`
3. 旧版数据不尝试反推单应用永久覆盖
4. 新版发布后，只有通过新版单应用菜单写入的记录，才会带 `force_*` 语义

原因：

- 旧版 UI 没有稳定的“单应用永久覆盖”模型
- 旧数据中无法安全区分批量应用和单应用操作
- 强行猜测会比统一收敛为全局物化更容易造成用户困惑

这意味着：本轮重构后，旧测试数据会优先按“跟随全局”解释，而不是按“永久单应用覆盖”解释。

## 图标包卸载与异常处理

### 当前图标包卸载

1. 清空 `current_icon_pack_package`
2. 对所有 `global_pack` 记录重新求值
3. 对 `force_pack` / `force_pack_custom` 记录做降级处理：
   - V1 建议直接回退为 `force_default`
   - 并提示“图标包已卸载，单独设置已回退为默认图标”
4. 批量刷新桌面与搜索

### 图标包未适配

若某应用未命中当前图标包：

- 在跟随全局场景下直接继续走后续回退逻辑
- 在单应用“使用当前图标包图标”场景下阻止操作

### 资源缺失或编码失败

- 当前应用跳过
- 记录日志，带上包名、组件名、drawableName
- 不影响其余应用处理

## 修改范围

### 重点修改

- `smali/com/smartisanos/home/settings/view/AppIconsSettingsActivity.smali`
- `smali/com/smartisanos/home/settings/view/AppIconsSettingsActivity$IconSettingsAdapter.smali`
- `smali/com/smartisanos/home/settings/icons/IconManager.smali`
- `smali/com/smartisanos/home/settings/icons/IconPackManager.smali`
- `smali/com/smartisanos/launcher/data/redirectIcon/RedirectIconInfo.smali`
- `smali/com/smartisanos/launcher/data/redirectIcon/RedirectIconDB.smali`
- 图标设置相关布局与文案

### 保持不动

- 桌面主渲染链路
- 搜索主消费逻辑
- 现有 iconData 刷新协议

## 验收标准

- 能独立选择“不使用图标包”或某个已安装图标包
- 改进版开关开启时，未单独设置的应用遵循 `改进版 > 图标包 > 默认`
- 改进版开关关闭时，未单独设置的应用遵循 `图标包 > 默认`
- 列表页每个应用都能明确显示当前来源
- 当前图标包未命中时，不再出现“已勾选但实际未适配”
- 单应用设置后，全局开关和图标包切换不影响该应用
- 单应用选择“跟随全局”后，能立即回到全局链路
- 当前图标包卸载后，不出现空白图标或脏状态

## 风险与取舍

- 最大工作量不在图标包解析，而在“旧二元页面”升级为“来源模型页面”
- 继续复用 `redirect_icons` 会让兼容逻辑略复杂，但能大幅降低渲染侵入
- 旧数据无法精确还原“历史上哪些是单应用永久覆盖”，本轮选择统一收敛为全局物化解释
- `从图标包中另选图标` 会增加资源浏览和选择成本，建议作为次级实现

## 开发顺序

1. 收口来源类型和全局优先级计算逻辑
2. 修正列表页状态展示，去掉错误勾选语义
3. 拆分设置首页入口，加入独立“设置图标包”
4. 重构应用列表页为“当前来源 + 单应用操作”
5. 接入 `force_*` 单应用覆盖模型
6. 完成旧数据兼容与图标包卸载回退
