# Smartisan Launcher Maintained Feature Migration Roadmap

## 背景

当前维护版主线基于旧桌面 `com.smartisanos.home`，目标是在新版 Android 和非锤子 ROM 上继续可安装、可启动、可维护。

同时，我们手上还有一个来自 R2 ROM 的新桌面 `LauncherSmartisanNew.apk` 维护仓库。新桌面功能更多，但系统耦合也更重。因此后续功能开发不适合简单切换基座，而应采用下面的策略：

- 旧桌面：继续作为发布主线
- 新桌面：作为功能参考仓库和实验来源
- 迁移原则：只迁移公开 Android API 可落地、系统耦合低、对兼容性收益明确的能力

## 当前主线判断

- 不建议切换到新桌面作为后续维护主线
- 建议继续基于旧桌面逐步修补和增强
- 新桌面适合用来：
  - 对照功能实现
  - 提取资源和展示逻辑
  - 分析哪些能力可以用通用 Android 方式重做

根因是新桌面明显依赖更多系统级权限和锤子私有能力，例如：

- `INTERACT_ACROSS_USERS_FULL`
- `INTERACT_ACROSS_USERS`
- `QUERY_ALL_PACKAGES`
- `queryIntentActivitiesAsUser`
- `startActivityAsUser`
- `DoppelgangerUtils`
- 多个锤子私有 `provider/service/permission`

这些能力不适合作为普通用户安装 APK 的长期基座。

## 迁移分级

### 一类：可直接迁移

这类能力通常不依赖系统签名，也不依赖锤子私有系统接口，适合继续优先迁移。

- 主题相关静态资产
  - 主题包名映射
  - 在线主题列表
  - 预览图资源
  - 主题展示文案
- 图标相关静态资源和接口策略
  - 旧接口优先
  - 新接口按包名兜底
  - 本地数据库回填旧桌面结构
- 部分纯资源/UI 层优化
  - 布局资源
  - 文案
  - 非系统耦合的展示细节

### 二类：需要改造后迁移

这类能力有实际价值，但不能直接照搬新桌面实现，必须按旧桌面的数据结构和兼容目标重做。

- 现代快捷方式支持
  - 新桌面已有 `PinShortcutActivity` 和 `LauncherApps / ShortcutInfo / PinItemRequest` 链路
  - 旧桌面目前主要还是老式 `INSTALL_SHORTCUT` 广播方案
  - 迁移方向应是：在旧桌面里新增现代快捷方式支持，而不是搬新桌面的整套实现
- 锁应用 / 密码确认
  - 旧桌面里已有部分历史入口和偏好字段
  - 新桌面有更完整的确认流程
  - 但新桌面依赖 `com.smartisanos.permission.LOCK_PASSWORD`
  - 如果后续要做，应按通用 Android 能力重新设计
- 大屏 / 长屏 / 横屏布局思路
  - 可以参考新桌面的布局分层和适配思路
  - 但不建议整体替换旧桌面的 view 架构
  - 更适合继续在旧桌面上做定点适配

### 三类：不建议迁移

这类能力高度依赖锤子私有系统或系统签名环境，不适合作为当前维护版主线目标。

- 双开 / 分身应用支持
  - 新桌面明确依赖 `userId`
  - 并且把 `userId=10` 当作双开模型的一部分
  - 同时依赖跨用户枚举和跨用户启动
  - 对非锤子 ROM 没有通用性
  - 当前主线已采用折中方案：只在搜索结果页补充显示并启动分身实例
  - 桌面首页、应用列表、数据库全链路 `userSerial` 改造仍不作为当前主线目标
- Sidebar / Launchpad / 桌面扩展服务
  - 新桌面包含额外 `provider/service`
  - 明显属于锤子系统生态内能力
- 新桌面的下载管理委托链路
  - 已经和私有通知权限、私有 Provider 绑定
  - 当前旧桌面的主题下载链路已经够用，不值得把系统耦合引进来
- Force Touch、桌面扩展调用、私有系统设置联动
  - 只要涉及私有权限或锤子专用入口，默认不进入主线

## 后续开发优先级

建议按下面顺序推进：

1. 继续维护旧桌面的兼容性和稳定性
2. 持续补充低耦合资源能力
   - 主题
   - 图标
   - 展示层优化
3. 评估并设计现代快捷方式支持
4. 评估是否要做通用版锁应用 / 隐藏应用
5. 明确放弃双开、Sidebar、Launchpad 一类高耦合能力

## 功能判断原则

后续每次从新桌面看到一个新功能时，统一先问下面三件事：

1. 是否依赖私有权限、系统签名或特权安装环境
2. 是否写死锤子特有模型
   - 例如固定 `userId=10`
   - 私有 `provider/service`
   - 私有桌面扩展能力
3. 是否可以用公开 Android API 在旧桌面中重新实现

判断规则：

- 如果第 1 条或第 2 条成立：不直接迁移
- 只有第 3 条明确成立：才进入迁移评估

## 建议的长期策略

为了让项目长期可控，建议坚持以下边界：

- 不把维护版目标改成“尽量还原锤子 ROM 全能力”
- 只做“普通安装包也能稳定运行”的功能增强
- 对于新桌面的能力，优先思考“是否能在旧桌面里重做”，而不是“能不能硬搬”

这样做的好处是：

- 兼容性边界清楚
- ROM 差异风险更低
- 后续每次修复和发版都更可控
- 项目不会逐步演变成“强依赖锤子系统环境”的半残分支

## 当前结论摘要

- 当前项目主线：继续维护旧桌面
- 当前参考仓库：新桌面仅作为功能矿和实验来源
- 当前最值得后续研究的新增能力：现代快捷方式支持
- 当前分身能力策略：保留搜索页折中支持，不推进桌面级全链路迁移
- 当前明确不建议投入主线的能力：双开、Sidebar、Launchpad、私有系统联动
